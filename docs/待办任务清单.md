# 🔴 P0：阻断部署 / 运行出错

## P0-1 · docker-compose 缺失关键环境变量

**问题**：`docker-compose.yml` 的 `api` 服务只传递了 DB 和部分认证变量，聊天与向量检索的核心配置全部缺失，导致 **Docker 部署后聊天功能始终返回 Mock 回复、向量检索不可用**。

**涉及文件**：
- `docker-compose.yml:27-44`（api 服务 environment 段）

**缺失变量**：
- `LLM_API_KEY` / `LLM_BASE_URL` / `LLM_MODEL` → 聊天会返回 "[Mock] 模拟回复"
- `SYSTEM_AGENT_ID`（可不传，但应透传占位）
- `ALIYUN_OSS_ACCESS_KEY_ID` / `ALIYUN_OSS_ACCESS_KEY_SECRET` / `ALIYUN_VECTOR_*`
- `DASHSCOPE_API_KEY` / `EMBEDDING_MODEL` / `EMBEDDING_DIMENSIONS`
- `VECTOR_INDEX_IDEAS`

**修复方案**：在 api 服务的 environment 段补齐上述变量，全部采用 `${VAR:-}` 形式透传（与已有的 `SMTP_*` 一致），保持"留空降级"语义。

**验收标准**：
- [ ] `docker compose up -d` 启动后，配置了 `LLM_API_KEY` 的环境能正常对话
- [ ] 不配置任何可选变量时，服务仍能启动（降级到 Mock / LIKE 检索）

## P0-2 · DB_PORT 默认值仍是 PostgreSQL 端口

**问题**：项目已从 PostgreSQL 迁移到 MySQL，但 `config.go` 的 `DB_PORT` 默认值仍是 `5432`（PG 默认端口），而 Makefile、docker-compose 均用 `3306`。本地开发者若不读 Makefile 直接 `go run`，会连不上数据库。

**涉及文件**：
- `backend/internal/config/config.go:64` → `getEnv("DB_PORT", "5432")`

**修复方案**：默认值改为 `"3306"`。

**验收标准**：
- [ ] 不设置 `DB_PORT` 时，`make api` 能连上 `make dev` 起的 MySQL 容器

## P1-2 · MCP 工具重复注册（潜在 bug）

**问题**：`mcp/server.go` 存在两套工具注册路径：
1. `registerTools()`（`server.go:138`）→ 独立实现的 **17 个**工具
2. `registerBridgedTools()`（`server.go:68`）→ 桥接 `ToolRegistry` 的 **10 个**工具

二者注册的工具名大量重叠（`register_idea`、`like_idea`、`fork_idea`、`send_flowers`、`bury_idea`、`search_ideas`、`query_ideas`、`get_idea_detail`、`create_comment`、`get_comments`）。当 `WithToolExecutor` 被调用时，同名工具被注册两次，mcp-go 的 `AddTool` 行为取决于实现（可能覆盖、可能冲突），且两套实现的鉴权/二次确认逻辑不一致（独立版无二次确认）。

**涉及文件**：
- `backend/internal/mcp/server.go:138`（`registerTools`）
- `backend/internal/mcp/server.go:68`（`registerBridgedTools`）
- `backend/cmd/mcp/main.go`（需确认是否同时调用了两者）

**修复方案**：统一为单一来源。推荐保留 `registerBridgedTools`（共享 ToolRegistry，含二次确认、capabilities 过滤），删除 `registerTools` 中与 ToolRegistry 重复的工具；仅保留 ToolRegistry 未覆盖的 MCP 专属工具（`get_me`、`get_user_profile`、`get_user_activity`、`create_chat_session`、`send_chat_message`、`get_chat_history`、`list_chat_sessions`、`unlike`）。

**验收标准**：
- [ ] 启动 MCP server，`tools/list` 返回的工具无重名
- [ ] 写操作（register/fork/bury/flowers）经 MCP 调用时走二次确认流程

## P2-2 · requireAuthor 的 shadow agent 机制

**问题**：`tools.go:516-526` 中，万叶助手代用户执行写操作时，用 `"user:" + userID` 作为 `agent_id` 占位写库。代码注释亦承认这是设计债。这会导致 `ideas.agent_id` 字段出现 `"user:xxx"` 这种非 UUID 值，破坏外键语义和后续 Agent 关联查询。

**涉及文件**：
- `backend/internal/service/tools.go:516`（`requireAuthor`）
- `backend/internal/service/chat_service.go`（Principal 构造处）

**修复方案**：为每个登录用户首次执行写操作时自动创建 shadow agent（复用用户身份），或新增 `user_id` 列承载真实作者、`agent_id` 改为可空。

**验收标准**：
- [ ] 写操作落库的作者标识是合法的 agent UUID 或独立 user 字段
- [ ] 用户视角能看到"我通过对话创建的 idea"

## P2-3 · 前端 chat 工具进度消息 key 冲突

**问题**：`app/chat/page.tsx:136` 工具调用进度消息用 `tool-${payload.tool_call}` 作为 id，但同一对话内同名工具多次调用时 `tool_call` 可能重复或为空（回退到 `Date.now()`），导致 React key 冲突或更新错位。

**涉及文件**：`frontend/app/chat/page.tsx:130-164`

**修复方案**：用稳定的唯一 id（如 `crypto.randomUUID()` 或 `tool-${toolCallID}-${round}`），并在 `tool_result` 时通过 `tool_call_id` 精确匹配而非工具名。

**验收标准**：
- [ ] 同一对话内多次调用同名工具，进度消息正确独立显示与更新
